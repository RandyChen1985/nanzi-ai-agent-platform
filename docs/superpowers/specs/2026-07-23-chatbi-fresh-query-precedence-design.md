# ChatBI 新查数与结果复用优先级设计

## 目标

修复“看看这周各智能体的调用情况，并可视化分析一下”被误判为上一轮结果分析的问题：请求包含新的业务主题和时间范围时，必须先进入 ChatBI 查数流程，再基于新结果可视化；只有明确指向上一轮结果的请求才允许短路复用。

## 方案

在 ChatBI 内层轮次分类中建立“新查数优先”门禁。复用上一轮结果前，先检查已有的新查数信号；若当前问题包含新的业务对象、时间范围、指标、明细或明确查数动作，则不执行 `RESULT_ANALYSIS` 早退。保留现有的“把刚才的结果画成图”复用路径。

上一轮结果是否存在仍由现有缓存机制提供；本次只修复分类优先级，不改变外层 DATA_QUERY/GENERAL 路由和反幻觉门禁，避免扩大行为范围。

## 流程

```text
当前请求
  ├─ 明确新查数信号 → NEW_DATA_QUERY → Schema/SQL/查询 → 可视化分析
  └─ 仅引用上一轮结果 → RESULT_ANALYSIS → 复用上一轮结果
```

## 验证场景

- “看看这周各智能体的调用情况，并可视化分析一下”必须是 `NEW_DATA_QUERY`，需要新鲜数据和 SQL。
- 上一轮是无关订单结果时，同一句请求仍必须是 `NEW_DATA_QUERY`。
- “把刚才的结果画成柱状图”仍是 `RESULT_ANALYSIS`，不重新查库。
- 现有“查询用户列表并可视化分析”回归测试继续通过。
